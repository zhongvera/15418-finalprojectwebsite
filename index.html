Overview

Summary

We plan on implementing and optimizing a continuous cellular automaton inspired by Conway's Game of Life, where each cell evolves according to a function computed using its neighbors' values. We will develop multiple parallel implementations and analyze how memory locality, computation intensity, and parallelization strategy affect performance. Our goal is to understand scalability limits and bottlenecks in stencil-style computations on multicore CPUs.
Context

Background

Conway's Game of Life is a discrete cellular automaton where binary-valued cells evolve based on simple rules applied to their neighbors. Continuous generalizations like SmoothLife and related work extend this by allowing cells to take on real-valued states (typically between 0 and 1), updated by a function over neighboring values. These models tend to produce richer, more stable, and more physically plausible dynamics.

In this project, we implement a continuous automaton on a 2D grid where each cell is updated based on a weighted aggregation of neighboring values combined with a nonlinear update function. This is similar to stencil operations in scientific computing and image processing. Unlike standard Game of Life, the continuous version introduces parameters like neighborhood radius and mathematical complexity, making it a meaningful workload for studying parallel performance.

Problem Statement

The Challenge

Although the computation is data-parallel, achieving high performance is nontrivial due to memory and locality constraints. Each cell update requires accessing neighboring values, leading to frequent memory accesses and potential cache inefficiencies. As the neighborhood radius grows, the working set grows and memory bandwidth may become the dominant bottleneck.

Different parallelization strategies can affect cache reuse and scaling behavior differently. While synchronization overhead is relatively low, speedup may be limited at higher thread counts due to memory bandwidth saturation and poor spatial locality. A further challenge is balancing computation and memory access: increasing computational complexity may improve arithmetic intensity but also changes the performance profile in nonobvious ways.

Infrastructure

Resources

We will use multicore CPU systems available through the GHC clusters and Bridges-2. The project will be implemented in C/C++ using OpenMP for thread-level parallelization, and potentially ISPC for SIMD optimization as a stretch goal. We are starting from scratch for the implementation, using the SmoothLife paper and related work as references for the continuous update rules. Standard clock-based timing tools will be used for benchmarking and bottleneck analysis.

Deliverables

Goals & Deliverables

Plan to Achieve (MVP)

Implement a correct serial version and at least two parallel OpenMP versions. Evaluate performance across different grid sizes and thread counts, and produce graphs showing runtime and speedup. Analyze how neighborhood radius affects performance and identify bottlenecks like memory bandwidth and cache behavior. Deliverables include working code, performance graphs, and a written analysis.

Hope to Achieve (Stretch)

Implement a SIMD-optimized version using ISPC or vector intrinsics. Experiment with different nonlinear update functions or larger neighborhoods to study how arithmetic intensity affects scalability. If the project goes well, compare CPU performance against a GPU baseline.

Rationale

Platform Choice

We chose multicore CPUs with OpenMP because the workload is naturally data-parallel and maps well to thread-based parallelism. OpenMP allows us to quickly implement and compare different decomposition strategies — such as row-parallel and tiled layouts — while maintaining fine-grained control over scheduling and thread behavior.

CPUs are also well-suited for studying cache locality and memory bandwidth effects, which are central to our research questions. The GHC cluster machines give us access to a consistent, well-characterized hardware environment with enough cores to observe meaningful scaling behavior.
